# Review: CLion/CMake CUDA Tooling & nvtx Fix

This document records the changes made to enable CUDA tooling/testing inside
CLion, and to fix the broken `nvtx` profiling integration, on branch
`test/cuda-rust`.

## 1. CLion CUDA build & test integration

**Problem:** CMake's native `project(... CUDA)` language cannot be used on
this host — `nvcc` 13.2 fails "CUDA compiler identification" because the
GCC 16 `libstdc++` headers use C++23 constructs nvcc's host-compiler front
end rejects (`type_traits`/`utility` errors, `char8_t` undefined, etc.), even
with `--allow-unsupported-compiler`. Switching the CUDA host compiler to
`clang++` hit an equivalent libstdc++ incompatibility.

**Fix:** `CMakeLists.txt` was rewritten to declare the project as `CXX` only
and drive `nvcc` through `add_custom_command` targets that compile each
`cu/*.cu` file straight to PTX (`nvcc -ptx ...`), mirroring the exact flags
already used by `build.rs` (`-std=c++17`, `-D__STRICT_ANSI__`,
`--allow-unsupported-compiler`, `--expt-relaxed-constexpr`,
`-Xcompiler -fno-builtin`). This sidesteps CMake's CUDA-language compiler
identification step entirely.

`enable_testing()` + CTest entries were added so CLion's built-in test
runner can drive everything:

| CTest name                       | What it runs                                                   |
|-----------------------------------|-----------------------------------------------------------------|
| `cargo_tests`                     | `cargo test --locked`                                            |
| `cargo_build_no_default_features` | `cargo build --locked --no-default-features`                     |
| `cargo_fmt_check`                 | `cargo fmt --check`                                              |
| `cargo_clippy_no_default`         | `cargo clippy --locked --no-default-features -- -D warnings`     |
| `cargo_build_bench_example`       | `cargo build --locked --features bench --example benchmark`      |
| `cuda_kernel_build`               | Builds the `cuda_kernels` target (PTX compile)                   |

Matching custom build targets (`cuda_kernels`, `cargo_test`,
`cargo_build_no_defaults`, `cargo_fmt`, `cargo_clippy`,
`cargo_bench_example`) are runnable directly from CLion's target
selector, independent of CTest.

**Verification:** `cmake --build cmake-build-debug --target cuda_kernels`
compiles all three `.cu` files to PTX; `ctest --output-on-failure` in
`cmake-build-debug` passes all registered tests.

**Known limitation:** if the host's GCC/Clang toolchain is upgraded, or a
CUDA toolkit update fixes the compiler-identification incompatibility, the
custom-command workaround should be revisited in favor of native
`project(... CUDA)` support.

**CLion note:** CLion defaults to the **Ninja** generator. A CLI
`cmake -B cmake-build-debug` without `-G Ninja` creates **Unix Makefiles**
and CLion will refuse to reload that directory. Fix: delete
`cmake-build-debug` and reconfigure with Ninja, or let CLion recreate the
profile. Do **not** use CLion “New Target” / `add_executable(... .cu)` —
that breaks CXX-only setup; use the `cuda_kernels` target instead.
`cuda_kernels` lists `cu/*.cu` and `cu/common.cuh` via `SOURCES` so the IDE
treats them as project files (clears “does not belong to any project target”).

**Bench note:** this crate has no `[[bench]]` / `#[bench]` targets. Use
`cargo test --features cuda -- --ignored` for the GPU unit test, and
`cargo run --example benchmark --profile bench --features bench,cuda` for
microbenchmarks. `cargo bench` only recompiles the lib tests under the
bench profile and will leave `#[ignore]` tests as ignored.

## 2. `.gitignore` fix — CMakeLists.txt was never tracked

**Problem:** `.gitignore` unconditionally excluded `CMakeLists.txt`. Since it
was *never* an already-tracked file, every edit made to it during CUDA/CMake
setup was silently dropped from `git status`/`git diff` and could not be
committed to the branch — hence "changes into the current branch" were
missing.

**Fix:** Removed the `CMakeLists.txt` line from `.gitignore`. Generated CMake
artifacts (`CMakeCache.txt`, `CMakeFiles/`, `cmake_install.cmake`, `Makefile`,
`cmake-build-*/`) remain ignored, since those are build outputs, not source.

Also added `benchmark_results.csv` / `benchmark_results.json` to the ignore
list — these are regenerated by `cargo run --example benchmark --features
bench` and were showing up as untracked noise.

## 3. `nvtx` was completely broken

Two independent bugs made `nvtx` non-functional:

1. **`Cargo.toml`**: declared `nvtx = { version = "1", features = ["macros"] }`.
   The `nvtx` crate (checked against docs.rs 1.3.0) has **no `macros`
   feature** — its `range_push!`/`range_pop!`/`range!`/`mark!` macros are
   unconditionally exported from the crate root. This alone made
   `cargo build`/`cargo test` fail dependency resolution for *every* build,
   CUDA or not.
2. **`src/gpu/kernel.rs`**: a stray edit had replaced the file's real imports
   (`GpuError`, `GpuResult`, `cust::Function`, `cust::Module`, `HashMap`)
   with only the `nvtx` import, and inserted an unrelated placeholder
   function with invalid syntax (`fn hot_path_example(...)`) calling an
   undefined, misspelled function (`lauunch_satsolver_kernel`). This broke
   compilation of the `cuda`-feature build independent of the `nvtx` issue.

**Fix:**
- `Cargo.toml`: `nvtx = "1"` (no invalid feature).
- `src/gpu/kernel.rs`: restored the original imports, removed the broken
  placeholder function, and wrapped `KernelModule::load()` in a real
  `range_push!("KernelModule::load")` / `range_pop!()` pair (delegating to a
  new private `load_inner()` for the actual logic) so nvtx now instruments
  real, load-bearing code instead of dead example code.

**Verification:**
- `cargo build --locked --no-default-features` — passes.
- `CUDA_NVCC=/usr/local/cuda/bin/nvcc cargo build --locked --features cuda` —
  passes, PTX kernels compile, `nvtx` compiles and links.
- `cargo test --locked` — unit/integration tests pass, 0 failed.
- `ctest` in `cmake-build-debug` — all registered tests pass (fmt, clippy,
  cargo test/build, bench example, cuda_kernels).

## 4. Benchmark GPU messaging

`--features bench` alone builds the stub GPU path (`gpu_stub`), so kernel
benches are compile-time disabled. Separately, `collect_gpu_info()` was a
TODO that always returned `None`, so the harness printed "No GPU detected"
even when a device was present under `--features bench,cuda`.

**Fix:** real `cust` device queries under `cuda`; honest log when built
without the feature (`use --features bench,cuda for GPU benchmarks`).

## 5. PTX `.version 8.5` broke sm_120 JIT

`build.rs` defaulted `MYELIN_PTX_VERSION` to the stub value `8.5` and always
rewrote nvcc's header. PTX ISA 8.5 does not support `.target sm_120`
(`ptxas` / driver return InvalidPtx / CUDA error 218). CI also pinned
`MYELIN_PTX_VERSION: "8.5"`.

**Fix:** leave nvcc's `.version` alone by default for non-Blackwell arches;
for `sm_12*` ensure a floor of `9.2`; only force-rewrite when
`MYELIN_PTX_VERSION` is set. CI now pins `9.2`.

## 6. CUDA GPU quality gate (local-first)

### Design intent: local is the source of truth

**This quality gate is primarily for local / self-hosted builds** on a machine
that actually has an NVIDIA GPU + matching driver + toolkit (e.g. ShipOfTheseus
with RTX 5080 / `sm_120`, driver ≥ 570, CUDA toolkit **13.2+**, local
baseline **13.3** when the tree is installed; driver UMD **13.x**).

Cloud GitHub-hosted runners are a poor fit for this gate:

- No Blackwell-class GPUs in standard `ubuntu-latest` pools
- Driver / toolkit / `sm_120` support is flaky or absent
- GPU minutes are expensive and queue-bound when available at all
- Feature-flagged stub builds already cover CPU CI without a device

So: **CPU checks always run in cloud CI; full CUDA runtime proof is local
(or optional self-hosted).** Do not treat a green cloud-only PR as “GPU
kernels validated end-to-end.”

### Local GPU quality gate (run on the GPU box)

Prereqs: `nvcc` on `PATH` (or `CUDA_NVCC=/usr/local/cuda/bin/nvcc`), working
`nvidia-smi`, driver that can load `sm_120` PTX. ShipOfTheseus default toolkit
is **13.3** via `/usr/local/cuda`.

#### Minimal path (proven green on branch `test/cuda-rust`)

```bash
cd ~/rmems/myelin-accelerator

# 1) Compile real CUDA path (nvcc → embedded PTX)
cargo build --lib --features cuda
# Expect: using nvcc: Build cuda_13.3...
# Expect: compiled spiking_network / vector_similarity / satsolver → sm_120, ptx>=9.2

# 2) Offline PTX assemble (ptxas needs a .ptx FILE + -o; -arch alone prints help)
#    CMake tree (after: cmake --build cmake-build-debug --target cuda_kernels):
ptxas -arch=sm_120 -o /tmp/sn.cubin cmake-build-debug/spiking_network.ptx
#    Or cargo OUT_DIR PTX:
# ptx_dir="$(ls -td target/debug/build/myelin-accelerator-*/out | head -1)"
# ptxas -arch=sm_120 -o /tmp/sn.cubin "$ptx_dir/spiking_network_sm_120.ptx"
# Success = no output, exit 0. Repeat for vector_similarity / satsolver if desired.

# 3) Runtime GPU unit test (JIT load + key symbols)
#    NOT cargo bench — this crate has no [[bench]] targets.
cargo test --features cuda -- --ignored
# Expect: test gpu::kernel::tests::test_load_kernels ... ok

# 4) Optimized microbenchmarks (device + launch)
cargo run --example benchmark --profile bench --features bench,cuda
# Expect: [bench] GPU: NVIDIA GeForce RTX 5080 (sm_120, … MB)
# Expect kernel rows: poisson_encode_4096, satsolver_extract_1024x256 (~5 µs mean)
# --features bench alone is stub path — not a GPU gate.
```

#### Extended path (CI-parity + CLion)

```bash
export CUDA_NVCC="${CUDA_NVCC:-/usr/local/cuda/bin/nvcc}"

cargo build --locked --features cuda --release
cargo clippy --locked --features cuda -- -D warnings
cargo test --locked --features cuda

# Cargo PTX shape (self-hosted CI does similar checks)
ptx_dir="$(ls -td target/debug/build/myelin-accelerator-*/out 2>/dev/null | head -n1)"
grep -q '^\.target sm_120' "$ptx_dir/spiking_network_sm_120.ptx"
grep -Eq '\.entry[[:space:]]+lif_step[[:space:]]*\(' "$ptx_dir/spiking_network_sm_120.ptx"

# CLion / Ninja profile (generator must be Ninja, not Unix Makefiles)
cmake -S . -B cmake-build-debug -G Ninja -DCUDA_NVCC="$CUDA_NVCC"
cmake --build cmake-build-debug --target cuda_kernels
ctest --test-dir cmake-build-debug --output-on-failure

# Optional Nsight
# nsys profile -o /tmp/myelin-timeline cargo run --example benchmark --profile bench --features bench,cuda
# ncu --set full -o /tmp/myelin-ncu cargo run --example benchmark --profile bench --features bench,cuda
```

### What “good” looks like

| Layer | Command | Proves |
|-------|---------|--------|
| Feature compile | `cargo build --lib --features cuda` | `nvcc` + `cust` + nvtx + PTX embed |
| Offline ISA | `ptxas -arch=sm_120 -o /tmp/x.cubin <file.ptx>` | PTX is valid for sm_120 |
| Runtime JIT | `cargo test --features cuda -- --ignored` | context + `KernelModule::load` on GPU |
| Launch + timing | `cargo run --example benchmark --profile bench --features bench,cuda` | kernels run (~5 µs class on 5080) |
| CMake PTX | `cmake --build … --target cuda_kernels` | CLion path matches `build.rs` flags |

**Do not use** `cargo bench` for this crate’s GPU gate — it has no `#[bench]` /
`[[bench]]` harness and will leave `#[ignore]` tests as ignored.

### Cloud CI vs local

| Job | Runner | GPU runtime? |
|-----|--------|--------------|
| Lint / CPU checks | GitHub-hosted `ubuntu-latest` | No — stub / no-default-features |
| `CUDA build [self-hosted] (sm_120)` | Labels `self-hosted,linux,x64,gpu,cuda` | Compile + clippy + `cargo test --features cuda` + PTX file checks — **only if a runner is online** |
| Local quality gate above | Developer workstation | Full runtime (ignored tests, bench, optional Nsight) |

CI’s cuda job **does not** pass `-- --ignored`, so `test_load_kernels` is not a
cloud gate unless someone adds that step on the self-hosted job. Prefer
keeping heavy runtime gates local so PRs from machines without GPUs stay
unblocked.
